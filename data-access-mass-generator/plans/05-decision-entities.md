# Plan: Decision Entities in the Mass Data Generator

## Overview

Add decision data to a configurable proportion of the applications generated by the mass generator.
For each chosen application, persist a `DecisionEntity` (overall decision), one `MeritsDecisionEntity`
per proceeding (individual merits outcome), and — when the overall decision is `GRANTED` — a
`CertificateEntity`. Finally, link the `DecisionEntity` back to the `ApplicationEntity` via the
`decision_id` foreign key, mirroring exactly what `ApplicationService.makeDecision()` does at runtime.

---

## Background: how `makeDecision` works

Reading `ApplicationService.makeDecision()` end-to-end, the full sequence for a decided application is:

1. `ApplicationEntity.isAutoGranted` is set to `true` or `false`.
2. A `DecisionEntity` is built with:
   - `overallDecision` — one of `GRANTED`, `PARTIALLY_GRANTED`, `REFUSED`
   - `meritsDecisions` — a `Set<MeritsDecisionEntity>`, one per proceeding
3. Each `MeritsDecisionEntity` holds:
   - `proceeding` — FK to `ProceedingEntity` (already persisted in the main loop)
   - `decision` — one of `GRANTED`, `REFUSED`  (`MeritsDecisionStatus`)
   - `reason` — free text, nullable
   - `justification` — free text, required
4. `MeritsDecisionEntity` rows are saved first via `meritsDecisionRepository.save()`.
5. `DecisionEntity` is saved via `decisionRepository.save()`.
6. When `overallDecision == GRANTED` and a certificate payload is present, a `CertificateEntity` is
   saved via `certificateRepository.save()`.
7. `application.setDecision(decision)` is called and the `ApplicationEntity` is saved again, writing
   the `decision_id` FK column.

The mass generator must replicate this sequence directly — it must **not** call `ApplicationService` or
any HTTP endpoint; it persists entities directly via repositories, exactly as the existing application
and proceedings loop does.

## Decision coverage

Not every generated application should have a decision — that would not reflect real data, where many
applications are still in progress. A `DECISION_RATE` constant (default `0.4`) controls the proportion.
The value is easy to change before a run.

```
40 % of applications → have a decision
  └─ 50 % of decided → REFUSED
  └─ 30 % of decided → GRANTED
  └─ 20 % of decided → PARTIALLY_GRANTED
```

These ratios are constants and trivial to adjust.

## New generator classes

Following the same pattern as the existing `Full*Generator` classes, two new generator classes are
needed inside `data-access-mass-generator`. No new model classes are required — `DecisionEntity`,
`MeritsDecisionEntity`, and `CertificateEntity` are all typed models already on the classpath.

### `FullMeritsDecisionGenerator`

Package: `uk.gov.justice.laa.dstew.access.massgenerator.generator.application`
Extends: `BaseGenerator<MeritsDecisionEntity, MeritsDecisionEntity.MeritsDecisionEntityBuilder>`

```java
public class FullMeritsDecisionGenerator
        extends BaseGenerator<MeritsDecisionEntity, MeritsDecisionEntity.MeritsDecisionEntityBuilder> {

    public FullMeritsDecisionGenerator() {
        super(MeritsDecisionEntity::toBuilder, MeritsDecisionEntity.MeritsDecisionEntityBuilder::build);
    }

    @Override
    public MeritsDecisionEntity createDefault() {
        return MeritsDecisionEntity.builder()
                .decision(faker.options().option(MeritsDecisionStatus.GRANTED, MeritsDecisionStatus.REFUSED))
                .justification(faker.lorem().sentence())
                .reason(faker.lorem().sentence())
                .build();
    }
}
```

The `proceeding` FK is **not** set here — it is injected in the runner loop using `.toBuilder()` so
that the proceeding already has a DB-assigned `id`.

### `FullCertificateGenerator`

Package: `uk.gov.justice.laa.dstew.access.massgenerator.generator.application`
Extends: `BaseGenerator<CertificateEntity, CertificateEntity.CertificateEntityBuilder>`

```java
public class FullCertificateGenerator
        extends BaseGenerator<CertificateEntity, CertificateEntity.CertificateEntityBuilder> {

    public FullCertificateGenerator() {
        super(CertificateEntity::toBuilder, CertificateEntity.CertificateEntityBuilder::build);
    }

    @Override
    public CertificateEntity createDefault() {
        return CertificateEntity.builder()
                .certificateContent(Map.of(
                        "certificateNumber", faker.regexify("[0-9]{4}/[0-9]{3} LEGAL HELP [A-Z]{3}"),
                        "issueDate", getRandomDate().toString(),
                        "validUntil", getRandomDate().plusYears(1).toString()
                ))
                .createdBy("mass-generator")
                .updatedBy("mass-generator")
                .build();
    }
}
```

`applicationId` is injected in the runner loop, not in `createDefault()`.

## Changes to `PersistedDataGenerator`

Register the three new repository mappings in `@PostConstruct` (alongside the existing ones):

```java
registerRepository(DecisionEntityGenerator.class,       DecisionRepository.class);
registerRepository(MeritsDecisionsEntityGenerator.class, MeritsDecisionRepository.class);
registerRepository(CertificateEntityGenerator.class,    CertificateRepository.class);
```

> `DecisionEntityGenerator`, `MeritsDecisionsEntityGenerator`, and `CertificateEntityGenerator` are the
> generator classes from `testUtilities` that are already used as the key type for repository lookups
> throughout `PersistedDataGenerator`. They do not need to be instantiated — they serve only as
> registration keys (the map key is the generator `Class<?>`, not an instance).
>
> The two new mass-generator generators (`FullMeritsDecisionGenerator`, `FullCertificateGenerator`) also
> need their own mappings so that `persist(generatorType, entity)` works correctly:
>
> ```java
> registerRepository(FullMeritsDecisionGenerator.class,  MeritsDecisionRepository.class);
> registerRepository(FullCertificateGenerator.class,     CertificateRepository.class);
> ```

## Changes to `MassDataGeneratorRunner`

### New constants

```java
private static final double DECISION_RATE = 0.4;   // 40 % of applications get a decision
```

### New generator fields

```java
private final FullMeritsDecisionGenerator meritsDecisionGenerator = new FullMeritsDecisionGenerator();
private final FullCertificateGenerator    certificateGenerator    = new FullCertificateGenerator();
```

### Decision block — inserted after step 5 (after proceedings are persisted)

The decision block runs inside the existing `for` loop, after `proceedingBatch` has been persisted and
the `ProceedingEntity` objects have DB-assigned UUIDs. The proceedings are fetched from the batch so
that their IDs are available.

```java
// 6. Optionally generate a decision for this application
if (faker.number().randomDouble(2, 0, 1) < DECISION_RATE) {

    DecisionStatus overallDecision = faker.options().option(
            DecisionStatus.REFUSED,        // 50 %
            DecisionStatus.REFUSED,
            DecisionStatus.REFUSED,
            DecisionStatus.GRANTED,        // 30 %
            DecisionStatus.GRANTED,
            DecisionStatus.PARTIALLY_GRANTED  // 20 %
    );

    // 6a. One MeritsDecisionEntity per proceeding
    Set<MeritsDecisionEntity> merits = new LinkedHashSet<>();
    for (ProceedingEntity pe : proceedingBatch) {
        MeritsDecisionEntity merit = meritsDecisionGenerator.createDefault(b -> b.proceeding(pe));
        persistedDataGenerator.persist(FullMeritsDecisionGenerator.class, merit);
        merits.add(merit);
    }

    // 6b. DecisionEntity
    DecisionEntity decision = DecisionEntity.builder()
            .overallDecision(overallDecision)
            .meritsDecisions(merits)
            .build();
    persistedDataGenerator.persist(FullMeritsDecisionGenerator.class /* key */, decision);
    // NOTE: use DecisionEntityGenerator.class as the key once registered, not FullMeritsDecisionGenerator

    // 6c. Certificate (GRANTED only)
    if (overallDecision == DecisionStatus.GRANTED) {
        CertificateEntity cert = certificateGenerator.createDefault(b -> b.applicationId(app.getId()));
        persistedDataGenerator.persist(FullCertificateGenerator.class, cert);
    }

    // 6d. Link decision back to application
    app.setDecision(decision);
    app.setIsAutoGranted(overallDecision == DecisionStatus.GRANTED);
    applicationRepository.save(app);
}
```

> `applicationRepository` is already available in `MassDataGeneratorRunner` as an `@Autowired` field —
> it is used via `persistedDataGenerator` — so either inject it directly or expose a thin method on
> `PersistedDataGenerator` to save an `ApplicationEntity` (preferred, to keep the runner thin).

## Sequence diagram

```
for each application (i = 0..count)
  │
  ├── generate ApplicationContent          (FullJsonGenerator)
  ├── persist ApplicationEntity            (ApplicationEntityGenerator)
  ├── link IndividualEntity                (native INSERT)
  ├── persist ProceedingEntity[]           (ProceedingsEntityGenerator)
  │
  └── if rand < DECISION_RATE
        │
        ├── for each ProceedingEntity
        │     └── persist MeritsDecisionEntity   (FullMeritsDecisionGenerator)
        │
        ├── persist DecisionEntity               (DecisionEntityGenerator key)
        │
        ├── if GRANTED
        │     └── persist CertificateEntity      (FullCertificateGenerator)
        │
        └── update ApplicationEntity.decision_id + isAutoGranted
```
