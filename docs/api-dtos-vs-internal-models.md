# Quick Reference: API DTOs vs Internal Models

## TL;DR

**YES**: Controllers should map API DTOs to internal command/query models.

**NO**: Don't pass OpenAPI-generated DTOs directly to use case services.

## The Pattern

```
API DTO (controller) → Internal Model (service) → Entity (repository)
```

## Code Examples

### ✅ CORRECT: Using Internal Models

```java
// 1. API DTO (OpenAPI-generated) - stays in controller
package uk.gov.justice.laa.dstew.access.model;

public class ApplicationCreateRequest {  // Generated by OpenAPI
    private UUID applyApplicationId;
    private String laaReference;
    private String applicationType;
    private Map<String, Object> applicationContent;
    // getters/setters
}
```

```java
// 2. Internal Command - used by service
package uk.gov.justice.laa.dstew.access.service.application.dto;

@Value
@Builder
public class CreateApplicationCommand {
    UUID applyApplicationId;
    String laaReference;
    String applicationType;
    Map<String, Object> applicationContent;
    
    public static CreateApplicationCommand from(ApplicationCreateRequest request) {
        return CreateApplicationCommand.builder()
            .applyApplicationId(request.getApplyApplicationId())
            .laaReference(request.getLaaReference())
            .applicationType(request.getApplicationType())
            .applicationContent(request.getApplicationContent())
            .build();
    }
}
```

```java
// 3. Controller - maps API DTO to internal model
package uk.gov.justice.laa.dstew.access.controller;

@RestController
@RequiredArgsConstructor
public class ApplicationController implements ApplicationApi {
    private final CreateApplicationService createApplication;
    
    @Override
    public ResponseEntity<Void> createApplication(
            ServiceName serviceName,
            ApplicationCreateRequest apiRequest) {  // API DTO from OpenAPI
        
        // Map to internal model
        CreateApplicationCommand command = CreateApplicationCommand.from(apiRequest);
        
        // Call service with internal model
        UUID id = createApplication.execute(command);
        
        URI uri = ServletUriComponentsBuilder
            .fromCurrentRequest()
            .path("/{id}")
            .buildAndExpand(id)
            .toUri();
        return ResponseEntity.created(uri).build();
    }
}
```

```java
// 4. Service - works with internal model
package uk.gov.justice.laa.dstew.access.service.application;

@Service
@RequiredArgsConstructor
@Transactional
public class CreateApplicationService {
    private final ApplicationRepository applicationRepository;
    private final ApplicationMapper applicationMapper;
    private final ApplicationValidations applicationValidations;
    
    public UUID execute(CreateApplicationCommand command) {  // Internal model
        // Business validation
        applicationValidations.validateCreateRequest(command);
        
        // Map to entity
        ApplicationEntity entity = applicationMapper.toEntity(command);
        
        // Business logic
        entity.setVersion(1);
        entity.setCreatedAt(Instant.now());
        
        // Persist
        ApplicationEntity saved = applicationRepository.save(entity);
        
        return saved.getId();
    }
}
```

```java
// 5. Mapper - handles internal model to entity conversion
package uk.gov.justice.laa.dstew.access.mapper;

@Mapper(componentModel = "spring")
public interface ApplicationMapper {
    
    // Internal command → Entity
    ApplicationEntity toEntity(CreateApplicationCommand command);
    
    // Entity → API DTO (for responses)
    Application toDto(ApplicationEntity entity);
    
    // Can also have: API DTO → Entity (for simple cases)
    ApplicationEntity toEntity(ApplicationCreateRequest request);
}
```

### ❌ INCORRECT: Passing API DTOs to Services

```java
// DON'T DO THIS
@RestController
public class ApplicationController {
    private final CreateApplicationService createApplication;
    
    @Override
    public ResponseEntity<Void> createApplication(
            ServiceName serviceName,
            ApplicationCreateRequest apiRequest) {
        
        // ❌ Passing API DTO directly to service
        UUID id = createApplication.execute(apiRequest);
        
        return ResponseEntity.created(buildUri(id)).build();
    }
}

@Service
public class CreateApplicationService {
    // ❌ Service coupled to API contract
    public UUID execute(ApplicationCreateRequest apiRequest) {
        // What happens when:
        // - OpenAPI spec changes?
        // - We add API v2?
        // - We add GraphQL?
        // This service breaks!
    }
}
```

## Real-World Example: Your Codebase

### Current (Before Refactoring)

```java
// ApplicationService.java - large service with many operations
@Service
public class ApplicationService {
    public UUID createApplication(ApplicationCreateRequest request) {
        // Uses API DTO directly
        ApplicationEntity entity = applicationMapper.toApplicationEntity(request);
        // ... business logic ...
        return applicationRepository.save(entity).getId();
    }
    
    public void updateApplication(UUID id, ApplicationUpdateRequest request) {
        // Uses API DTO directly
        // ... business logic ...
    }
    
    // ... 8 more methods ...
}
```

**Problems**:
- Service coupled to OpenAPI-generated DTOs
- Can't support API v2 without changing service
- Hard to test without constructing API DTOs

### Proposed (After Refactoring)

```java
// CreateApplicationService.java - focused service
@Service
@RequiredArgsConstructor
public class CreateApplicationService {
    private final ApplicationRepository applicationRepository;
    private final ApplicationMapper applicationMapper;
    private final ApplicationValidations applicationValidations;
    
    public UUID execute(CreateApplicationCommand command) {  // Internal model
        applicationValidations.validateCreateRequest(command);
        ApplicationEntity entity = applicationMapper.toEntity(command);
        entity.setVersion(1);
        ApplicationEntity saved = applicationRepository.save(entity);
        return saved.getId();
    }
}

// UpdateApplicationService.java - separate focused service
@Service
@RequiredArgsConstructor
public class UpdateApplicationService {
    private final ApplicationRepository applicationRepository;
    private final ApplicationMapper applicationMapper;
    private final DomainEventService domainEventService;
    
    public void execute(UUID id, UpdateApplicationCommand command) {  // Internal model
        ApplicationEntity entity = applicationRepository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException(id));
        
        applicationMapper.updateEntityFromCommand(entity, command);
        applicationRepository.save(entity);
        
        domainEventService.recordUpdate(id);
    }
}
```

```java
// Controller maps API → Internal
@RestController
@RequiredArgsConstructor
public class ApplicationController implements ApplicationApi {
    private final CreateApplicationService createApplication;
    private final UpdateApplicationService updateApplication;
    
    @Override
    public ResponseEntity<Void> createApplication(
            ServiceName serviceName,
            ApplicationCreateRequest apiRequest) {
        CreateApplicationCommand command = CreateApplicationCommand.from(apiRequest);
        UUID id = createApplication.execute(command);
        return ResponseEntity.created(buildUri(id)).build();
    }
    
    @Override
    public ResponseEntity<Void> updateApplication(
            ServiceName serviceName,
            UUID id,
            ApplicationUpdateRequest apiRequest) {
        UpdateApplicationCommand command = UpdateApplicationCommand.from(id, apiRequest);
        updateApplication.execute(id, command);
        return ResponseEntity.noContent().build();
    }
}
```

**Benefits**:
- Services work with internal models
- API changes only affect controllers
- Easy to test services with POJOs
- Can add API v2 or GraphQL easily

## When to Skip Internal Models

For very simple operations, you can skip internal models:

```java
// Simple read-by-id - OK to skip internal model
@Service
@RequiredArgsConstructor
public class GetApplicationByIdService {
    private final ApplicationRepository repository;
    private final ApplicationMapper mapper;
    
    public Application execute(UUID id) {
        ApplicationEntity entity = repository.findById(id)
            .orElseThrow(() -> new ResourceNotFoundException(id));
        return mapper.toDto(entity);  // Direct entity → API DTO
    }
}
```

But for operations with business logic, **always use internal models**:

```java
// Complex operation - MUST use internal model
@Service
@RequiredArgsConstructor
public class AssignCaseworkerService {
    private final ApplicationRepository applicationRepository;
    private final CaseworkerRepository caseworkerRepository;
    private final DomainEventService domainEventService;
    
    public void execute(AssignCaseworkerCommand command) {  // Internal model
        // Business validation
        if (command.getApplicationIds().size() > 50) {
            throw new ValidationException("Cannot assign more than 50 applications at once");
        }
        
        // Business logic
        CaseworkerEntity caseworker = caseworkerRepository.findById(command.getCaseworkerId())
            .orElseThrow(() -> new ResourceNotFoundException(command.getCaseworkerId()));
        
        List<ApplicationEntity> applications = 
            applicationRepository.findAllById(command.getApplicationIds());
        
        applications.forEach(app -> {
            app.setCaseworker(caseworker);
            app.setUpdatedAt(Instant.now());
        });
        
        applicationRepository.saveAll(applications);
        
        // Domain events
        command.getEventHistory().forEach(event -> 
            domainEventService.recordEvent(event));
    }
}
```

## Testing Examples

### ✅ WITH Internal Models - Easy Testing

```java
@ExtendWith(MockitoExtension.class)
class CreateApplicationServiceTest {
    @Mock
    private ApplicationRepository repository;
    
    @Mock
    private ApplicationMapper mapper;
    
    @InjectMocks
    private CreateApplicationService service;
    
    @Test
    void shouldCreateApplication() {
        // Simple POJO - easy to construct
        CreateApplicationCommand command = CreateApplicationCommand.builder()
            .applyApplicationId(UUID.randomUUID())
            .laaReference("LAA-ABC123")
            .applicationType("LEGAL_AID")
            .applicationContent(Map.of("key", "value"))
            .build();
        
        ApplicationEntity entity = new ApplicationEntity();
        entity.setId(UUID.randomUUID());
        
        when(mapper.toEntity(command)).thenReturn(entity);
        when(repository.save(entity)).thenReturn(entity);
        
        UUID result = service.execute(command);
        
        assertThat(result).isEqualTo(entity.getId());
        verify(repository).save(entity);
    }
}
```

### ❌ WITHOUT Internal Models - Complex Testing

```java
@ExtendWith(MockitoExtension.class)
class ApplicationServiceTest {
    @Mock
    private ApplicationRepository repository;
    
    @InjectMocks
    private ApplicationService service;
    
    @Test
    void shouldCreateApplication() {
        // Complex API DTO - harder to construct, coupled to API
        ApplicationCreateRequest apiRequest = new ApplicationCreateRequest();
        apiRequest.setApplyApplicationId(UUID.randomUUID());
        apiRequest.setLaaReference("LAA-ABC123");
        apiRequest.setApplicationType("LEGAL_AID");
        apiRequest.setApplicationContent(Map.of("key", "value"));
        // ... many more fields ...
        
        // Test coupled to API structure
        // If API changes, test breaks
    }
}
```

## Summary

| Aspect | API DTOs Only | With Internal Models |
|--------|---------------|----------------------|
| **Coupling** | ❌ Services coupled to API | ✅ Services independent of API |
| **API Evolution** | ❌ Break services when API changes | ✅ Change only controllers |
| **Testing** | ❌ Complex API DTO construction | ✅ Simple POJO construction |
| **Multi-Interface** | ❌ Hard to add GraphQL/gRPC | ✅ Easy to add new interfaces |
| **Business Logic** | ❌ Mixed with API concerns | ✅ Pure business logic |
| **Versioning** | ❌ Hard to support v1 and v2 | ✅ Easy to support multiple versions |

## Decision: Always Use Internal Models

For the pragmatic refactoring, we will:

✅ **Create internal command/query models** for all use cases with business logic  
✅ **Map API DTOs to internal models** in controllers  
✅ **Use case services work exclusively with internal models**  
✅ **Skip internal models only for trivial read-by-id operations**

This keeps business logic independent of API contracts while maintaining a pragmatic, low-complexity approach.
